\title{A Multi-Dimensional Priority Model for Heterogeneous DAG Workloads in Distributed Computing Continuums}

\author{
\IEEEauthorblockN{Authors}
\IEEEauthorblockA{Affiliation}
}

\maketitle

\begin{abstract}
Distributed meta-orchestration platforms coordinate workloads across heterogeneous infrastructure spanning cloud, edge, and HPC systems. These platforms must schedule diverse workloads---from safety-critical mine seismology to autonomous AI agents to best-effort research analytics---on shared executor pools. Existing schedulers rely on a single integer priority, which conflates urgency, importance, fairness, and domain-specific value into one dimension, leading to priority inversions and starvation.

We analyze eight realistic workload types---including self-modifying agentic AI workflows and real-time IoT edge scenarios requiring preemption---identify common priority dimensions, and evaluate six scheduling approaches. Based on this analysis, we propose a hybrid priority model combining lexicographic urgency classes with weighted multi-dimensional scoring within each class. The model supports static, dynamic, and DAG-dependent priority dimensions while providing structural guarantees for safety-critical workloads, preemption on resource-constrained executors, and multi-tenant fairness.
\end{abstract}

\begin{IEEEkeywords}
distributed scheduling, DAG workflows, priority models, computing continuums, meta-orchestration, agentic AI, preemption
\end{IEEEkeywords}

\section{Introduction}
\label{sec:introduction}

Computing continuums~\cite{dustdar2023computing} integrate heterogeneous resources---cloud data centers, HPC clusters, edge devices, and IoT infrastructure---into unified execution environments. Meta-orchestration platforms such as ColonyOS~\cite{colonyos} coordinate workloads across these environments by acting as brokers between users who declare \emph{what} to compute and distributed executors that know \emph{how} to execute on their specific platforms.

Workloads in these systems are expressed as directed acyclic graphs (DAGs) of processes, where each process specifies a function to execute, its arguments, resource requirements, and dependencies on other processes. The scheduler assigns waiting processes to available executors based on priority.

The fundamental challenge is that these platforms serve radically different workloads concurrently: safety-critical mine seismology, satellite surveillance, \emph{agentic AI systems that dynamically modify their own DAGs}, multi-tenant ETL pipelines with SLA deadlines, infrastructure automation, cross-environment HPC training, large-scale parameter sweeps, and real-time IoT edge systems requiring preemption. Each workload has its own notion of what ``important'' means, yet they all compete for shared executor resources.

Current systems assign priority as a single integer at submission time. This approach fails in three fundamental ways:

\begin{enumerate}
    \item \textbf{Priority is multi-dimensional.} A seismic event's importance depends on sensor count, mine region, and event classification---not a single number. An AI agent's priority depends on its iteration depth, token budget consumption, and goal criticality.
    \item \textbf{Priority is dynamic.} An ETL pipeline's urgency increases as its SLA deadline approaches. An agentic workflow's resource profile is unknown at submission---it depends on LLM decisions at runtime. A drug screening compound's priority depends on docking results from an earlier DAG step.
    \item \textbf{Priority is contextual.} A cloud burst workflow's urgency depends on another workflow's queue depth. On resource-constrained edge devices, an emergency anomaly response must preempt running lower-priority monitoring processes.
\end{enumerate}

In this paper, we make the following contributions:

\begin{itemize}
    \item We present eight realistic use cases spanning the diversity of workloads in computing continuums, including self-modifying agentic AI workflows and real-time IoT with preemption (Section~\ref{sec:usecases}).
    \item We extract common priority dimensions and identify six requirements that any priority model must satisfy (Section~\ref{sec:analysis}).
    \item We evaluate six alternative scheduling approaches against these requirements (Section~\ref{sec:methods}).
    \item We propose a hybrid priority model combining lexicographic urgency classes with weighted multi-dimensional scoring, and show how it instantiates for each use case (Section~\ref{sec:model}).
\end{itemize}

\section{Background}
\label{sec:background}

\subsection{Computing Continuums and Meta-Orchestration}

A computing continuum provides a unified abstraction over heterogeneous infrastructure. Unlike traditional orchestrators tied to specific platforms (Kubernetes for cloud, SLURM for HPC), a meta-orchestrator operates \emph{above} platform-specific schedulers, coordinating workloads across environments without directly controlling the underlying resources.

In ColonyOS, the core abstractions are:

\begin{itemize}
    \item \textbf{Colony:} A distributed runtime environment comprising networked executors.
    \item \textbf{Executor:} A worker process that connects to the colony from any network location, pulls process assignments, executes them, and reports results. Executors can run on cloud VMs, HPC nodes, edge devices, or any Internet-connected system.
    \item \textbf{Process:} A unit of work defined by a function specification, transitioning through states: WAITING $\rightarrow$ RUNNING $\rightarrow$ SUCCESS or FAILED.
    \item \textbf{ProcessGraph:} A DAG of processes with dependency edges. Child processes wait until all parent processes complete successfully.
    \item \textbf{WorkflowSpec:} A declarative specification of a ProcessGraph, defining processes and their dependencies by name.
\end{itemize}

Executors request work by calling \texttt{Assign()}, which returns the highest-priority waiting process matching the executor's type and capabilities. This pull-based model means the scheduler's priority function directly determines which work gets done first.

\subsection{DAG Scheduling}

DAG scheduling is well-studied in parallel computing~\cite{kwok1999static}. Classical algorithms (HEFT~\cite{topcuoglu2002performance}, CPOP) focus on minimizing makespan on a fixed set of processors. Our setting differs in several ways: executors are heterogeneous and dynamically available, DAGs can be modified at runtime (processes inserted between existing nodes), and multiple independent DAGs compete for shared resources with different urgency levels.

\subsection{Current Priority Mechanism}

In the current system, each process has an integer \texttt{Priority} field set at submission time. The scheduler computes a \texttt{PriorityTime} value:

\begin{equation}
    \text{PriorityTime} = \text{Priority} \times \Delta t + \text{SubmissionTime}
\end{equation}

where $\Delta t$ is a constant scaling factor. The process with the lowest PriorityTime is assigned first. This creates a FIFO ordering within the same priority level, with higher-priority processes jumping ahead. While simple, this mechanism cannot express the multi-dimensional, dynamic, context-dependent priority requirements we identify in our use cases.

\subsection{Dynamic DAG Modification}

ColonyOS supports runtime modification of ProcessGraphs through the \texttt{AddChild(insert=true)} mechanism. This allows an executor to insert new processes \emph{between} existing nodes in a running DAG, effectively growing the workflow at runtime. This capability is central to the agentic AI use case (Section~\ref{sec:uc3}), where an LLM-powered agent iteratively expands its plan, and distinguishes our setting from classical DAG schedulers that assume a fixed graph structure.

\subsection{UC1: Mine Seismology (Continuous Stream)}
\label{sec:uc1}

A seismic monitoring network at the LKAB Kiruna iron ore mine has 12 geophones at four underground levels (1165--1450m depth). Seismogram data streams via RabbitMQ; a new processing DAG is submitted every 60~seconds with three stages: (1)~\textbf{P-wave picking} (12~parallel)---STA/LTA detection with AIC onset refinement on each geophone; (2)~\textbf{Event classification} (1~process)---a CNN classifier groups picks into events (blast, rockburst, noise); (3)~\textbf{3D location} ($N$~parallel)---grid search with Simplex refinement using a layered velocity model. Each DAG has $12 + 1 + N$ processes.

\textbf{Priority:} Rockbursts must preempt blasts (safety-critical). Events with more triggered sensors indicate larger magnitudes and should be prioritized---but sensor count is only known \emph{after} classification (runtime-dependent). Active production levels are higher priority than exploration zones.

\subsection{UC2: Satellite Surveillance (Event-Triggered, Multi-User)}
\label{sec:uc2}

Multiple organizations define surveillance jobs monitoring geographic areas using Sentinel-2 imagery. When new imagery is available, a 4-step DAG is spawned: download $\rightarrow$ preprocess (cloud mask, co-register) $\rightarrow$ change detection $\rightarrow$ notify. Five concurrent jobs from Swedish agencies cover illegal logging, highway construction, flooding, wildfire recovery, and coastline erosion, each using domain-specific algorithms (BFAST Monitor, coherence change, MNDWI, dNBR, NDWI+Otsu).

\textbf{Priority:} Detection type determines urgency (wildfire $>$ flooding $\gg$ erosion monitoring). Multiple jobs trigger simultaneously from the same satellite pass. Cross-organization fairness is required.

\subsection{UC3: Agentic AI Workflows (Self-Modifying DAG)}
\label{sec:uc3}

Users submit natural-language goals to an LLM-powered agentic executor~\cite{colonyos}. The system constructs and \emph{iteratively extends} a DAG at runtime:

\begin{enumerate}
    \item A skeleton DAG is submitted: \texttt{agent\_query} $\rightarrow$ \texttt{llm\_plan} $\rightarrow$ \texttt{plan\_handler} $\rightarrow$ \texttt{completed}.
    \item The plan handler parses the LLM's output and \emph{inserts} tool steps and a reflect node into the DAG between existing nodes using \texttt{AddChild(insert=true)}.
    \item After tool steps execute, the reflect node evaluates results. If more work is needed, additional steps and another reflect are inserted.
    \item This loop continues until the goal is achieved or a resource limit is reached.
\end{enumerate}

This produces a \emph{self-modifying DAG}: the workflow decides its own shape at runtime (Fig.~\ref{fig:agentic_dag}). Each iteration adds 2--5 processes, and typical workflows run 3--10 iterations, producing DAGs of 10--50 processes from a 4-process skeleton.

Multiple users submit goals concurrently, each producing an independent ProcessGraph that grows dynamically and unpredictably. Three representative workloads run simultaneously: a data science agent analyzing soil moisture datasets, a DevOps agent writing an MQTT-to-InfluxDB integration, and a student agent requesting explanations of scheduling concepts.

\textbf{Priority requirements:}
\begin{itemize}
    \item \textbf{Unknown DAG size:} Resource consumption is unpredictable at submission time---a simple question may resolve in one iteration, while a complex coding task may require ten.
    \item \textbf{Iteration depth decay:} An agent on its 15th iteration is consuming shared resources with potentially diminishing returns. Priority should decay with iteration depth to prevent runaway agents from monopolizing executors.
    \item \textbf{Token/cost budget:} LLM API calls are expensive. An agent approaching its token budget limit may warrant priority adjustment---either boosted (finish before cutoff) or reduced (diminishing value).
    \item \textbf{Goal criticality:} A production agent handling a customer request should outprioritize an exploratory research agent, but this is a user-declared dimension, not derivable from the DAG.
    \item \textbf{Tool contention:} When multiple agents need the same external tool (database query, API call), priority determines which agent proceeds first.
    \item \textbf{Implicit priority propagation:} Prioritizing an agent's current step implicitly prioritizes all future work it will generate. Unlike fixed DAGs, where the scheduler can see all downstream processes, an agentic DAG's future is opaque.
\end{itemize}

\subsection{UC4: Multi-Tenant ETL (SLA-Driven)}
\label{sec:uc4}

A shared data platform serves three organizations with different SLAs: \textbf{Gold} (5-min deadline, bank fraud detection, 10~processes), \textbf{Silver} (60-min, logistics route optimization, 7~processes), and \textbf{Bronze} (24h, university bibliometrics, 5~processes). All pipelines compete for the same \texttt{etl-worker} executors.

\textbf{Priority:} SLA tier determines base priority. Urgency escalates as deadlines approach. Bronze must not be starved---fairness across tenants is required despite priority differences.

\subsection{UC5: Cloud Burst (Meta-Level Infrastructure)}
\label{sec:uc5}

When the seismic queue (UC1) exceeds on-premise capacity, a cloud burst DAG is triggered: assess queue $\rightarrow$ estimate capacity $\rightarrow$ provision VMs (Hetzner) $\rightarrow$ install K3s $\rightarrow$ deploy executors + transfer data (ColonyFS) $\rightarrow$ start $\rightarrow$ monitor costs $\rightarrow$ monitor queue $\rightarrow$ teardown. Budget cap: EUR~50.

\textbf{Priority:} Derives from \emph{another workflow's} queue state (externally-dependent). Creates a feedback loop: burst capacity drains the queue, reducing its own urgency. Budget constraints impose a ceiling independent of priority.

\subsection{UC6: Cloud-to-HPC LLM Training (Cross-Environment)}
\label{sec:uc6}

A cloud platform user fine-tunes Llama~3.1 8B for Swedish legal text. Two parallel preparation tracks (data: validate $\rightarrow$ tokenize $\rightarrow$ upload; model: download $\rightarrow$ convert Megatron $\rightarrow$ upload) converge at HPC training on Dardel (2~nodes $\times$ 4$\times$A100, TP=4, PP=2). Results transfer back via ColonyFS to MLflow registry and vLLM deployment.

\textbf{Priority:} The DAG crosses environment boundaries with long data transfers. HPC processes may queue in SLURM for hours. Priority must span two independent scheduling domains.

\subsection{UC7: Drug Discovery Parameter Sweep (Deadline-Driven Funnel)}
\label{sec:uc7}

A pharmaceutical company screens 25 drug candidates against SARS-CoV-2 Mpro. Phase~1: prepare target $\rightarrow$ dock all 25 (AutoDock Vina) $\rightarrow$ 25 short MD simulations (10ns, GROMACS) $\rightarrow$ rank. Phase~2: top~3 candidates get 100ns extended MD $\rightarrow$ regulatory report. Hard 48-hour deadline.

\textbf{Priority:} Compounds are prioritized by ML-predicted binding affinity (better candidates get GPU time first). Phase~2 processes outprioritize remaining phase~1. Deadline proximity escalates urgency.

\subsection{UC8: Water Treatment IoT (Real-Time Edge with Preemption)}
\label{sec:uc8}

A water treatment plant (Norrvatten, northern Stockholm) monitors quality across four stages using edge gateways with 1--2 executor slots each. Normal monitoring DAGs (collect $\rightarrow$ validate $\rightarrow$ aggregate $\rightarrow$ dashboard) run every 30s, producing 16 normal-priority processes per cycle. When a sensor breaches a threshold (chlorine $<$~0.2~mg/L, turbidity $>$~4~NTU), an emergency DAG is submitted: emergency collect $\rightarrow$ classify severity $\rightarrow$ actuate response + alert operators (parallel) $\rightarrow$ notify SCADA $\rightarrow$ log incident.

\textbf{Priority:} Emergency processes must \emph{preempt} running normal-priority processes---not merely jump the queue---because edge gateways lack spare capacity. Hard real-time constraint: anomaly to actuator within 2~seconds. Mixed criticality levels share the same resource-constrained executors.

\section{Analysis of Priority Dimensions}
\label{sec:analysis}

Examining the eight use cases reveals that priority is determined by multiple dimensions. We classify these along two axes: \emph{when} the value is known and \emph{where} it comes from.

\subsection{Temporal Classification}

\textbf{Static dimensions} have values fixed at submission time: SLA tier (UC4), detection type (UC2), compound predicted binding affinity (UC7), anomaly severity (UC8), user tier and goal criticality (UC3).

\textbf{Dynamic dimensions} change over time without new information: deadline proximity (UC4, UC7), process wait time (all), latency budget remaining (UC8), token budget consumption (UC3).

\textbf{Runtime-dependent dimensions} only become known when intermediate DAG steps complete: triggered sensor count (UC1, after classification), candidate ranking (UC7, after short MD), DAG size and iteration depth (UC3, determined by LLM at runtime), confirmed severity after cross-validation (UC8).

\textbf{Externally-dependent dimensions} derive from state outside the DAG: queue depth of another workload (UC5), HPC queue wait time (UC6), budget remaining (UC5), sensor readings breaching thresholds (UC8).

\subsection{Source Classification}

\textbf{Process-intrinsic:} Values from the process's own specification---function name, arguments, resource requirements, executor type.

\textbf{DAG-structural:} Values from the process's position in its DAG---depth, number of downstream dependents, DAG completion percentage, iteration depth in agentic loops (UC3).

\textbf{Parent-derived:} Values from the output of parent processes---classifier results, intermediate scores, LLM plan decisions (UC3).

\textbf{System-state:} Values from the scheduling system---queue depths, executor utilization, elapsed time.

\textbf{User/tenant metadata:} Values from the process submitter---SLA tier, organizational role, budget, user tier.

\subsection{Requirements}

From the use cases, we derive six requirements for a priority model:

\begin{enumerate}
    \item[\textbf{R1}] \textbf{Structural safety guarantees.} Safety-critical work (UC1 rockbursts, UC2 wildfire alerts, UC8 anomaly response) must be guaranteed to preempt non-critical work, independent of parameter tuning.

    \item[\textbf{R2}] \textbf{Multi-dimensional scoring.} Priority must be expressible as a function of multiple domain-specific dimensions (sensor count, mine region, event type, compound affinity, anomaly severity, agent iteration depth).

    \item[\textbf{R3}] \textbf{Dynamic re-evaluation.} Priority must be recalculable when parent processes complete (UC1), deadlines approach (UC4), external state changes (UC5), or DAGs grow dynamically (UC3).

    \item[\textbf{R4}] \textbf{Multi-tenant fairness.} Lower-priority tenants must receive a minimum guaranteed share of resources, preventing starvation (UC4 Bronze tier, UC3 student agents).

    \item[\textbf{R5}] \textbf{Interpretability.} Scheduling decisions must be explainable---critical for safety-critical systems (UC1, UC8), SLA compliance auditing (UC4), and understanding why an AI agent was deprioritized (UC3).

    \item[\textbf{R6}] \textbf{Preemption support.} On resource-constrained executors (UC8 edge gateways), higher-priority processes must be able to preempt running lower-priority processes, not merely jump the queue.
\end{enumerate}

\section{Alternative Scheduling Methods}
\label{sec:methods}

We evaluate six approaches against the requirements.

\subsection{Weighted Sum Model}

The priority of a process $p$ at time $t$ is:

\begin{equation}
    \pi(p, t) = \beta(p) + \sum_{i=1}^{n} w_i \cdot d_i(p, G, S, t)
    \label{eq:wsm}
\end{equation}

where $\beta(p)$ is a base priority, $d_i$ are dimension functions, and $w_i$ are user-configured weights.

\textbf{Strengths:} Simple, interpretable, each dimension independently tunable, $O(n)$ per evaluation. \textbf{Weaknesses:} Assumes dimension independence. Dimensions are commensurable: with misconfigured weights, a Bronze process can outprioritize a rockburst alert, violating R1. \textbf{Assessment:} R2, R3, R5. Partially R4. Fails R1, R6.

\subsection{Rule-Based / Production Systems}

An ordered set of condition-action rules; the first match determines priority. \textbf{Strengths:} Handles non-linear dimension interactions; readable and auditable (R1, R5). \textbf{Weaknesses:} Rule explosion with growing dimensions. Dynamic rules require complex meta-rules. \textbf{Assessment:} R1, R5. Partially R2. Fails R3, R4, R6.

\subsection{Lexicographic / Hierarchical Priority}

Priority is a tuple $\pi(p) = (c_1(p), c_2(p), \ldots, c_k(p))$ compared left to right. \textbf{Strengths:} Safety is structural---if urgency class is $c_1$, safety-critical work \emph{cannot} be outprioritized (R1). Used in Linux CFS, Kubernetes PriorityClasses~\cite{kubernetes2023priority}. \textbf{Weaknesses:} Rigid within levels; cannot express smooth trade-offs. \textbf{Assessment:} R1, R5. Partially R2, R3, R6.

\subsection{Constraint-Based Optimization}

Scheduling formulated as: $\underset{x}{\text{maximize}} \sum_{p \in P} U(p) \cdot x_p$ subject to resource, fairness, and deadline constraints. \textbf{Strengths:} Most expressive---can encode R1--R6 as constraints. \textbf{Weaknesses:} ILP at every scheduling decision does not scale. Requires global view of all pending work. Decisions are opaque (R5 violated). \textbf{Assessment:} R1--R4, R6 in theory. Fails R5. Impractical at scheduling-time granularity.

\subsection{Multi-Level Feedback Queue (MLFQ)}

Processes start in the highest queue and demote after consuming their time quantum. Used in OS schedulers. \textbf{Strengths:} Naturally penalizes long-running processes, providing fairness (R4) without explicit configuration. Handles the agentic AI case: agents that consume many iterations naturally demote. \textbf{Weaknesses:} Queue levels are a coarse proxy for priority---cannot express domain-specific dimensions (R2 poorly satisfied). Gaming-susceptible (processes can yield to avoid demotion). Safety guarantees require a reserved top queue with admission control. \textbf{Assessment:} Partially R1, R4. Fails R2, R3, R5, R6.

\subsection{Machine Learning Methods}

\textbf{RL:} Agents learn scheduling policies by optimizing reward functions (DeepRM~\cite{mao2016resource}, Decima~\cite{mao2019learning}). \textbf{GNNs:} Encode DAG topology to prioritize bottleneck processes. \textbf{Learned weights:} Automate $w_i$ tuning via supervised learning on historical outcomes. \textbf{Strengths:} Can discover non-obvious strategies; GNNs handle DAG-aware scheduling; adapt to changing workloads. \textbf{Weaknesses:} When a rockburst alert is deprioritized, ``the neural network assigned 0.73'' is not an acceptable explanation. Training data requirements. Generalization failures on new workload types. \textbf{Assessment:} R2--R4. Fails R1, R5, R6.

\section{Proposed Model}
\label{sec:model}

We propose a two-level priority model: lexicographic \emph{urgency classes} provide structural guarantees (R1, R5), while a weighted multi-dimensional \emph{scoring function} within each class provides expressiveness (R2, R3). Fairness constraints (R4) are enforced orthogonally. Preemption (R6) is triggered by urgency class transitions. Fig.~\ref{fig:model} illustrates the model architecture.

\centering

    node distance=0.4cm,
    box/.style={rectangle, draw, rounded corners=3pt, minimum width=3.2cm, minimum height=0.55cm, font=\small, align=center},

    arr/.style={-{Stealth[length=2mm]}, thick},
    lbl/.style={font=\tiny, text=gray},
]
    % Urgency classes
    \node[classbox, fill=red!20] (safety) {\textsc{Safety}};
    \node[classbox, fill=orange!20, below=0.2cm of safety] (deadline) {\textsc{Deadline}};
    \node[classbox, fill=blue!10, below=0.2cm of deadline] (normal) {\textsc{Normal}};
    \node[classbox, fill=gray!10, below=0.2cm of normal] (bg) {\textsc{Background}};

    % Scoring
    \node[box, fill=yellow!10, right=1.5cm of safety, yshift=-0.6cm] (score) {$\sigma(p,t) = \beta + \sum w_i d_i$};

\subsection{Urgency Classes}

Every process is assigned to exactly one urgency class $C \in \{$\textsc{Safety}, \textsc{Deadline}, \textsc{Normal}, \textsc{Background}$\}$, ordered by strict dominance:

\begin{equation}
    \textsc{Safety} \succ \textsc{Deadline} \succ \textsc{Normal} \succ \textsc{Background}
\end{equation}

A process in a higher class is \emph{always} scheduled before a process in a lower class. Class assignment uses promotion rules:

\begin{equation}
    C(p, t) = \max_{r \in R} \{ c_r \mid \phi_r(p, G, S, t) = \text{true} \}
\end{equation}

where $R$ is the set of promotion rules, $c_r$ is the target class, and $\phi_r$ is a boolean predicate. The default class is \textsc{Normal}. Examples:

\begin{itemize}
    \item \textsc{Safety}: \texttt{event\_type = ``rockburst'' $\wedge$ sensor\_count $\geq$ 6} (UC1); \texttt{anomaly\_severity = ``critical''} (UC8).
    \item \textsc{Deadline}: \texttt{deadline\_remaining $<$ 0.1 $\times$ deadline\_total} (UC4, UC7); \texttt{latency\_budget $<$ 500ms} (UC8).
    \item \textsc{Background}: \texttt{sla = ``Bronze'' $\wedge$ no\_deadline} (UC4); \texttt{agent\_iteration $>$ 10 $\wedge$ tier = ``student''} (UC3).
\end{itemize}

These rules are \emph{auditable}: a human can inspect the rule set and verify that safety-critical work will always be classified correctly. This is a structural guarantee, not a statistical one.

\subsection{Within-Class Scoring}

Within each urgency class, processes are ordered by:

\begin{equation}
    \sigma(p, t) = \beta(p) + \sum_{i=1}^{n} w_i \cdot d_i(p, G, S, t)
    \label{eq:score}
\end{equation}

where $d_i$ are \emph{dimension functions} mapping a process and its context to a normalized value in $[0, 1]$:

\begin{itemize}
    \item $d^{\text{kwarg}}_{k}(p) = f(\text{kwarg}(p, k))$: From process arguments.
    \item $d^{\text{parent}}_{k}(p, G) = f(\text{output}(\text{parent}(p, G), k))$: From parent output (available after parent completes).
    \item $d^{\text{dag}}(p, G) = |\{q \in G : \text{state}(q) = \text{SUCCESS}\}| / |G|$: DAG completion fraction (``momentum'').
    \item $d^{\text{wait}}(p, t) = \min(1, (t - t_{\text{submit}}(p)) / \tau)$: Normalized wait time (anti-starvation).
    \item $d^{\text{deadline}}(p, t) = 1 - \min(1, (t_{\text{deadline}}(p) - t) / t_{\text{total}}(p))$: Deadline proximity.
    \item $d^{\text{system}}_{k}(S) = f(\text{state}(S, k))$: System state (queue depth, budget).
    \item $d^{\text{iter}}(p, G)$: Iteration depth in an agentic loop---the number of reflect-plan-execute cycles completed. Used with a negative weight for priority decay.
\end{itemize}

The transform function $f$ normalizes raw values to $[0, 1]$ using linear, categorical, exponential (for deadlines), or logarithmic (for anti-starvation) mappings.

\subsection{Re-evaluation Triggers}

The effective priority $\Pi(p, t) = (C(p, t), \sigma(p, t))$ is re-evaluated on:

\begin{enumerate}
    \item \textbf{Parent completion:} Enables parent-derived dimensions and can trigger class promotion (UC1: classification reveals rockburst; UC3: LLM plan determines next steps).
    \item \textbf{Timer tick:} Updates time-dependent dimensions ($d^{\text{wait}}$, $d^{\text{deadline}}$) and can trigger deadline-based promotion.
    \item \textbf{System state change:} Queue depth or executor availability changes trigger re-evaluation of $d^{\text{system}}$.
    \item \textbf{DAG modification:} When new processes are inserted (UC3), the iteration depth and DAG size dimensions are updated for all affected processes.
\end{enumerate}

\subsection{Fairness Constraints}

Fairness is enforced orthogonally to priority scoring:

\begin{equation}
    \forall j \in \text{Tenants}: \frac{\text{assigned}(j, \Delta t)}{\text{total\_assigned}(\Delta t)} \geq s_j
\end{equation}

When a tenant falls below its minimum share $s_j$, its next process is scheduled regardless of score. This is particularly relevant for UC3 (agentic AI) and UC4 (multi-tenant ETL), where high-priority agents or Gold-tier pipelines could otherwise monopolize executors.

\subsection{Preemption}

When a process is promoted to \textsc{Safety} and the target executor has no idle slots, the scheduler preempts the lowest-priority running process. Preemption is:

\begin{itemize}
    \item Only triggered \emph{across} class boundaries---never within the same class.
    \item Configurable per executor type: \textbf{preemptive} (edge gateways, UC8), \textbf{non-preemptive} (cloud/HPC with spare capacity), or \textbf{cooperative} (checkpoint before yielding).
\end{itemize}

The preempted process is marked WAITING with a \texttt{preempted} flag and resumed (from checkpoint or restart) when capacity is available.

\subsection{Effective Priority}

The complete priority is a tuple compared lexicographically:

\begin{equation}
    \Pi(p, t) = \big(C(p, t),\ \sigma(p, t)\big)
\end{equation}

The scheduler selects the waiting process with the highest $\Pi$ matching the requesting executor's type.

\subsection{Instantiation Examples}

We illustrate model instantiation for four use cases.

\subsubsection{UC1: Mine Seismology}

\begin{lstlisting}[language=,caption={Seismic processing priority.}]
promotion_rules:
  SAFETY: event_type="rockburst" AND count>=6
dimensions:
  sensor_count: parent_output("classify","count")
    linear(3,12), weight=5
  mine_region: kwarg("region")
    categorical(production:1, development:0.7), w=3
  event_type: parent_output("classify","class")
    categorical(rockburst:1, blast:0.5), w=4
\end{lstlisting}

The \texttt{sensor\_count} and \texttt{event\_type} dimensions are parent-derived: available only after classification completes, triggering re-evaluation and potential \textsc{Safety} promotion.

\subsubsection{UC3: Agentic AI}

\begin{lstlisting}[language=,caption={Agentic AI priority with iteration decay.}]
promotion_rules:
  BACKGROUND: iteration>10 AND tier="student"
dimensions:
  goal_criticality: kwarg("goal_tier")
    categorical(production:1, research:0.5,
                student:0.2), weight=6
  iteration_depth: dag_iteration_count()
    linear(0,15), weight=-4  # negative = decay
  token_budget_remaining: kwarg("budget_pct")
    linear(0,100), weight=2
  dag_momentum: dag_completion()
    linear(0,1), weight=3
fairness:
  min_share: {production:0.5, research:0.3,
              student:0.1}
\end{lstlisting}

The \emph{negative} weight on \texttt{iteration\_depth} causes priority to decay as the agent iterates, preventing runaway agents from monopolizing resources. The \texttt{token\_budget\_remaining} dimension gives a slight boost to agents with budget left (they can still produce useful work). The \textsc{Background} promotion rule demotes student agents that exceed 10 iterations, ensuring they don't block production workloads.

\section{Discussion}
\label{sec:discussion}

\subsection{The Agentic AI Scheduling Challenge}

Use Case~3 introduces a scheduling problem qualitatively different from the other seven: the workload \emph{decides its own shape at runtime}. In all other use cases, the scheduler can inspect the full DAG at submission time (or after a fixed growth phase). An agentic DAG is opaque---prioritizing an agent's current process implicitly prioritizes all future work it will generate, which is unknowable.

This creates a tension between two principles: (1)~\emph{sunk cost protection}---an agent that has completed 8 successful iterations has generated value and should be allowed to finish; and (2)~\emph{diminishing returns}---an agent that has iterated 15 times without completion is likely stuck and should yield to other workloads. Our model handles this through the iteration depth dimension with a negative weight (priority decay) combined with DAG momentum (sunk cost protection). The balance between these two forces is configurable via weights.

More broadly, the rise of agentic AI workloads challenges the assumption that DAG scheduling is about \emph{fixed graphs}. Future scheduling models must account for self-modifying DAGs as a first-class concept, including resource budgets that bound total agent consumption regardless of priority.

\subsection{Composability}

Priority models compose naturally when multiple workload types share a colony. Each workload type defines its own dimensions and weights, but the urgency classes provide a shared framework for inter-workload ordering. A colony administrator defines promotion rules and fairness constraints; workload developers define within-class scoring.

\subsection{Relationship to Classical Scheduling}

The model subsumes classical policies as special cases: \textbf{FIFO} (no dimensions, equal base priority), \textbf{EDF} (one deadline dimension), \textbf{SJF} (one dimension on estimated execution time), \textbf{Fair Share} (fairness constraints with equal shares), \textbf{Fixed Priority with Preemption} (urgency classes with preemptive policy). This generality suggests the model can serve as a unifying framework.

\subsection{Role of Machine Learning}

While we argued against ML as a standalone approach, ML complements the model in two ways: (1)~\textbf{Weight tuning}---learning $w_i$ from historical outcomes while preserving the interpretable structure; (2)~\textbf{Dimension prediction}---predicting values not yet available (e.g., likely sensor count before classification completes in UC1, or expected iteration count for an agentic workflow in UC3).

\subsection{Preemption Trade-offs}

UC8 demonstrates that queue-ordering is insufficient on resource-constrained executors. Preemption introduces costs (restart overhead, checkpointing), but these are acceptable when the alternative violates hard real-time safety constraints. Limiting preemption to cross-class boundaries bounds its frequency. A key open question: if a process in a 90\%-complete DAG is preempted, should the DAG momentum bonus apply when rescheduled? We argue yes.

\subsection{Inter-Workflow Dependencies}

UC5 illustrates a pattern where one workflow's priority derives from another's state. Our model handles this through $d^{\text{system}}$ dimensions, but does not formally model inter-DAG edges. Priority propagation across DAG boundaries---where a high-priority Workflow~B causes Workflow~A's blocking process to inherit urgency---is an open problem that requires preventing circular dependencies and unbounded propagation.

\subsection{Failure Recovery}

A failed process in a partially completed DAG presents a priority dilemma: boost (protect DAG investment), reduce (may fail again), or escalate (approaching deadline). The model's re-evaluation mechanism handles deadline escalation naturally. A \emph{retry count} dimension could address the boost/reduce trade-off, with deployment-specific weight polarity.

\subsection{Limitations}

The model assumes urgency classes can be cleanly separated. In practice, class boundaries may be politically contested. The model requires explicit configuration (promotion rules, dimensions, weights, fairness shares), imposing operational burden. Not all workloads are preemptible---GPU kernels and database transactions cannot be cleanly interrupted. The per-executor-type preemption policy mitigates this but requires correct classification by operators.

For agentic AI workloads specifically, the iteration depth dimension with a negative weight is a heuristic. Some agents legitimately require many iterations (complex coding tasks), and a fixed decay penalizes them. An adaptive decay based on progress indicators (e.g., fraction of subtasks completed) would be more principled but harder to configure.

\section{Related Work}
\label{sec:related}

\textbf{DAG scheduling.} HEFT~\cite{topcuoglu2002performance} and CPOP prioritize tasks by upward rank for single-DAG makespan optimization on fixed processors. Our work addresses multi-DAG scheduling on dynamic executors with heterogeneous priority semantics.

\textbf{Multi-criteria scheduling.} Braun et al.~\cite{braun2001comparison} compare heuristics for heterogeneous computing. Most multi-criteria approaches target makespan and utilization; we focus on priority as a user-facing concern with safety and fairness requirements.

\textbf{Fair scheduling.} DRF~\cite{ghodsi2011dominant} provides max-min fairness for multi-resource environments. YARN~\cite{vavilapalli2013yarn} implements capacity scheduling with hierarchical queues and preemption. Our fairness constraints are simpler but interact with a richer priority model.

\textbf{Mixed-criticality systems.} Vestal~\cite{vestal2007preemptive} introduced mixed-criticality scheduling for real-time systems, where tasks have different criticality levels and the system must guarantee timing for the highest level. Burns and Davis~\cite{burns2019mixed} survey the field. Our urgency classes are analogous to criticality levels, extended to DAG workflows in distributed settings rather than periodic tasks on uniprocessors.

\textbf{Kubernetes scheduling.} Kubernetes PriorityClasses~\cite{kubernetes2023priority} implement preemption based on priority levels. Our model extends this with multi-dimensional scoring within each level and DAG-aware re-evaluation.

\textbf{RL for scheduling.} DeepRM~\cite{mao2016resource} and Decima~\cite{mao2019learning} apply deep RL to resource management and DAG scheduling. These sacrifice interpretability, which we argue is essential for safety-critical deployments.

\textbf{Workflow systems.} Apache Airflow, Prefect, and Dagster support DAG workflows with basic priority. None provide multi-dimensional dynamic priority, DAG-aware re-evaluation, or runtime DAG modification.

\textbf{Agentic AI orchestration.} LangGraph, CrewAI, and AutoGen orchestrate LLM agent workflows but treat scheduling as a local concern. Our model addresses the multi-agent, multi-user scheduling problem where agents compete for shared resources with different criticality levels.

\textbf{Computing continuums.} Dustdar et al.~\cite{dustdar2023computing} survey the computing continuum vision. Our work contributes a concrete scheduling model for the heterogeneous workloads that arise in these environments.

\section{Conclusion}
\label{sec:conclusion}

We presented eight diverse workload use cases in computing continuums---including self-modifying agentic AI workflows and real-time IoT with preemption---and demonstrated that single-integer priority is insufficient for scheduling heterogeneous DAG workloads. Through systematic analysis, we identified six requirements and showed that no existing method satisfies all six.

Our hybrid priority model combines lexicographic urgency classes with weighted multi-dimensional scoring. The urgency classes provide auditable structural guarantees for safety-critical work and serve as triggers for preemption on resource-constrained executors. The scoring function provides expressive, domain-specific ordering within each class, including novel dimensions such as agent iteration depth decay for agentic AI workloads. Fairness constraints prevent starvation independently of priority scores.

The model subsumes classical scheduling policies as special cases, yet each component is independently configurable and interpretable. We identified inter-workflow dependencies, failure recovery, and adaptive agent decay as open problems. Future work includes implementation in ColonyOS, empirical evaluation with concurrent workloads, learned weight tuning, and formal analysis of preemption overhead on edge deployments.
